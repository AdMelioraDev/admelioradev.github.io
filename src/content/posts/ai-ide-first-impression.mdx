AI 기반 IDE인 Windsurf를 처음 마주했을 때의 심정도 비슷했다.
'이러다 우리 모두 실업자 되는 거 아닌가' 하는 불안감으로 시작했지만 — 사실 지금도 그 생각이 완전히 사라지지는 않았다 — AI 어시스턴트와 함께 난제를 하나씩 해결해나가며 새로운 가능성을 발견했다.
마치 화성에서 살아남아 결국에는 지구로 무사히 돌아간 와트니처럼 우리도 이 도전을 기회로 만들어낼 수 있다는 확신이 들었다.

## 왜 AI IDE 여야만 하는가

대다수의 개발자들이 그렇겠지만 나 또한 이미 현업에서 AI 를 유용하게 사용하고 있다. 기본 개발 환경은 Visual Studio 를 사용하면서 궁금한 것이 있거나 새로운 정보가 필요할 때면 ChapGPT 로부터 유용한 정보를 얻곤 한다.
이 정도만 해도 구글링과 Stack Overflow에 의존했던 예전보다는 훨씬 나은 환경이긴 하지만, 여전히 몇 가지 불편한 점이 있었다. 예를 들자면:

- <span className="text-[#0070f3]">컨텍스트 단절</span>
  - 개발 컨텍스트를 매번 설명해야 한다. 프로젝트의 구조, 사용 중인 프레임워크, 의존성 등을 AI가 알지 못해서 대화할 때마다 다시 설명해야 했다.
- <span className="text-[#0070f3]">수동적인 코드 관리</span>
  - 코드 변경이 실시간으로 반영되지 않는다. AI가 제안한 코드를 직접 복사하여 붙여넣고, 여러 파일을 수정할 때는 일일이 찾아가며 수정해야 했다.
- <span className="text-[#0070f3]">제한된 런타임 정보</span>
  - 에러의 실시간 컨텍스트를 파악하지 못한다. 런타임 에러가 발생했을 때 스택 트레이스나 변수 상태 등 실행 환경의 전체적인 맥락을 AI에게 전달하기 어려웠다.

결국 대화창이라는 인터페이스의 한계 때문에 일정 이상의 심도 있는 상호 작용이 불가능했다는 것이 가장 큰 불만이었다.
때문에, 단순한 대화창을 넘어 IDE 자체에서 일어나는 모든 일을 파악할 수 있는 AI 도구가 필요하다는 생각을 하게 되었다.

## Windsurf와의 첫 만남

자, 그렇다면 어떤 도구를 골라야할까? 제일 먼저 떠오른 후보는 Cursor였다.
우연히 읽게 된 블로그 덕에 알게 된 Cursor는 Visual Studio Code에서 파생된 프로젝트라는 점, 내 개발 환경에서 손쉽게 사용할 수 있다는 점 때문에 가장 먼저 떠오른 선택지였다.
다음 후보는 Windsurf였다. Cursor와 마찬가지로 Visual Studio Code 기반의 IDE로, Cursor보다는 후발 주자이지만 그 덕분에 더 경제적인 구독료로 사용할 수 있었다.
AI IDE를 처음 사용하는 입장에서 제대로 된 비교는 어려웠기에 후발 주자이니만큼 좀 더 혁신적인 부분이 있지 않을까 하는 막연한 기대로 Windsurf를 선택하게 되었다.

![Windsurf IDE 개발 환경](/images/posts/windsurf-ide.png)
*Windsurf 실행 화면. Visual Studio 였다면 솔루션 탐색기가 있었을 자리에 Cascade 대화창이 있는 것이 보인다.*

이렇게 시작된 Windsurf와의 여정에서 발견한 장단점들을 정리해보았다.

## 장점

- 신속한 프로토타이핑
  - 아이디어를 빠르게 구현할 수 있다
  - 기본적인 보일러플레이트 코드를 자동으로 생성
- 컨텍스트 이해
  - 프로젝트의 전체적인 구조를 이해하고 있어 일관된 코드 생성
  - 기존 코드베이스와의 통합이 자연스러움
- 학습 기회
  - AI가 제안하는 구현 방식에서 새로운 패턴을 배울 수 있음
  - 모범 사례를 자연스럽게 접할 수 있음

## 개선이 필요한 부분

- 의도 전달의 정확성
  - 때로는 의도를 정확히 전달하는 것이 코드를 직접 작성하는 것보다 어려울 수 있음
  - 복잡한 로직의 경우 여러 번의 대화가 필요
- 제어력
  - 세세한 구현을 직접 제어하기가 어려울 수 있음
  - 때로는 과도하게 복잡한 솔루션을 제안하기도 함

## 앞으로의 가능성

AI IDE는 분명 개발 방식의 큰 변화를 가져올 것이다.
특히 프로토타이핑이나 반복적인 작업에서 큰 효율성 향상을 기대할 수 있다.
하지만 이는 개발자를 대체하는 것이 아니라, 개발자의 생산성을 높이는 도구로 발전할 것으로 보인다.

앞으로 더 많은 경험을 쌓으면서 이 도구를 어떻게 더 효과적으로 활용할 수 있을지 계속 고민해보려 한다.
